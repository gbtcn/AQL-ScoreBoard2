# AGENTS.md

## このプロジェクトの目的

このプロジェクトは、既存の `AQL Score Board` の HTML / CSS / JavaScript 実装を元に、改修するためのベース仕様である。

Codex は、既存実装の見た目・操作感・競技ロジックを理解したうえで、ユーザーから指示される差分を反映して Web ページを改修すること。

既存コードは以下の構成である。

```text
AQL-ScoreBoard/
├─ html/
│  └─ aql-sb.html
├─ css/
│  └─ aql-sb.css
└─ js/
   ├─ aql-sb.js
   └─ syncscroll.jsj
```

## 重要方針

- 既存実装から読み取れる仕様を尊重する。
- ユーザーから明示された差分指示がある場合は、その差分を優先する。
- 仕様変更の際は、画面構造・競技ロジック・履歴表示・勝敗判定への影響を必ず確認する。
- 大きな変更を行う前に、何を変えるのか、なぜ変えるのかを簡潔に説明する。
- JavaScript の処理は、可能であれば DOM 直書き依存を減らし、状態管理と描画を分離する。
- ただし、最初から過度に大規模な設計変更をしない。現行仕様を再現できることを優先する。

## 改修開始前の時点の画面全体の概要

この Web ページは、AQL の団体戦スコアボードである。

画面には以下を表示する。

1. タイトル
2. 全体操作ボタン
3. 試合設定
4. 現在の問題番号と共通操作
5. チーム A のスコアボード
6. チーム B のスコアボード
7. 各チームの履歴表

## 画面構造

### タイトル

画面上部に `AQL Score Board` と表示する。

現行実装では `#title` に対して `font-size: 48px` が指定されている。

### 全体操作

`#settings` 内に以下のボタンがある。

| ボタン | 表示 | 動作 |
|---|---|---|
| `btn-reset` | `reset` | 全状態をリセットする |
| `btn-refresh` | `refresh` | 現在の DOM 状態を元に再計算する |

### 試合設定

`#config` 内に以下の設定項目がある。

| 項目 | 初期値 | 説明 |
|---|---:|---|
| `winning-pt` | `200` | 勝利に必要なチーム得点 |
| `max-of-questions` | `40` | 規定問題数 |
| `infinity` | 未チェック | チェック時は問題数上限なし |

`winning-pt` と `max-of-questions` は、フォーカスアウト時に入力値を正規化・検証する。

- 全角数字は半角数字に変換する。
- 前後の空白は除去する。
- 正の整数のみ有効とする。
- 不正な入力の場合は、直前の有効値に戻す。
- 有効値は `escape-value` 属性にも保持する。

`infinity` をチェックした場合、`max-of-questions` は無効化され、表示値は `no limit` になる。チェックを外すと、直前の有効値に戻る。

### 共通表示・共通操作

`#common` 内に以下がある。

| 要素 | 表示 / 初期値 | 説明 |
|---|---|---|
| `secret-counter` | `1` | 内部的な現在問題番号。画面上は非表示 |
| `number-of-done` | `1` | 画面表示用の現在問題番号 |
| `through` | `Through` | スルー扱いで問題数を進める |
| `show-hide-counter` | `Hide Count` | 問題番号の表示 / 非表示を切り替える |
| `history-show-hide` | `Hide History` | 履歴表の表示 / 非表示を切り替える |

問題番号は「N問目」の形で表示する。

`show-hide-counter` を押すと以下のように動作する。

- ボタン表示が `Hide Count` のとき、押すと `Show Count` になる。
- カウント非表示時、画面上の問題番号は `?` になる。
- カウント非表示時、履歴表の問題番号列も非表示になる。
- 再度表示すると、実際の問題番号を表示する。

`history-show-hide` を押すと以下のように動作する。

- ボタン表示が `Hide History` のとき、押すと `Show History` になり、履歴エリアを非表示にする。
- ボタン表示が `Show History` のとき、押すと `Hide History` になり、履歴エリアを表示する。

## チーム構造

チームは A / B の 2 チームである。

各チームは 5 人のプレイヤーを持つ。

| チーム | プレイヤー ID |
|---|---|
| A | `a1`, `a2`, `a3`, `a4`, `a5` |
| B | `b1`, `b2`, `b3`, `b4`, `b5` |

現行表示では、チーム A は左から `A5, A4, A3, A2, A1` の順に並ぶ。チーム B は左から `B1, B2, B3, B4, B5` の順に並ぶ。

この左右の並びは、対戦表示上の左右対称性を意図している可能性があるため、変更する場合は注意する。

## チームスコアボード

各チームには以下の行がある。

1. 勝敗結果表示行
2. チーム名入力行
3. チーム得点行
4. プレイヤー名入力行
5. プレイヤー個人得点行
6. プレイヤー誤答数行
7. 正解 / 誤答ボタン行

### 勝敗結果表示行

勝敗が決まると、チームごとに以下を表示する。

- 勝利チーム: `WIN`
- 敗北チーム: `LOSE`
- 引き分け: 両チーム `DRAW`
- 継続中: 空欄

勝利チームのチーム得点セルはオレンジ背景になる。

### チーム名

各チーム名はテキストボックスで編集できる。

初期値は以下である。

| チーム | 初期値 |
|---|---|
| A | `A` |
| B | `B` |

### プレイヤー名

各プレイヤー名はテキストボックスで編集できる。

初期値は `A1` から `A5`、`B1` から `B5` である。

### 個人得点

各プレイヤーの個人得点の初期値は `1` である。

正解すると、そのプレイヤーの個人得点が `+1` される。

誤答すると、そのプレイヤーの個人得点は `1` に戻る。

個人得点セルをダブルクリックすると、手動修正できる。

- 入力は正の整数のみ有効。
- 全角数字は半角数字に変換する。
- 不正な入力の場合はアラートを表示し、変更しない。

### 誤答数

各プレイヤーの誤答数は、以下の 3 状態で表す。

| 表示 | 意味 |
|---|---|
| 空欄 | 誤答なし |
| `✕` | 1 誤答 |
| `✕✕` | 2 誤答。解答権なし / 封鎖状態 |

誤答セルをダブルクリックすると、手動修正できる。

有効な入力は以下のみである。

- 空欄
- `✕`
- `✕✕`

それ以外は不正入力としてアラートを表示し、変更しない。

### 正解ボタン

各プレイヤーに `〇` ボタンがある。

正解ボタン押下時の動作は以下である。

1. 試合がすでに終了している場合は何もしない。
2. 対象プレイヤーが `✕✕` の場合は何もしない。
3. 対象プレイヤーの個人得点を `+1` する。
4. 問題番号を `+1` する。
5. 履歴表に正解履歴 `〇` を追加する。
6. 全体を再計算する。

### 誤答ボタン

各プレイヤーに `✕` ボタンがある。

誤答ボタン押下時の動作は以下である。

1. 試合がすでに終了している場合は何もしない。
2. 対象プレイヤーの誤答数を進める。
   - 空欄 → `✕`
   - `✕` → `✕✕`
   - `✕✕` → 何もしない
3. 相手チームに `✕✕` のプレイヤーがいる場合、そのプレイヤーを `✕` に戻す。
4. 対象プレイヤーの個人得点を `1` に戻す。
5. 問題番号を `+1` する。
6. 履歴表に誤答履歴 `✕` を追加する。
7. 全体を再計算する。

## 得点計算

チーム得点は、所属する 5 人の個人得点の積である。

```text
チーム得点 = プレイヤー1得点 × プレイヤー2得点 × プレイヤー3得点 × プレイヤー4得点 × プレイヤー5得点
```

初期状態では全員の個人得点が `1` のため、チーム得点は `1` である。

例：

```text
個人得点: 2, 1, 3, 1, 2
チーム得点: 2 × 1 × 3 × 1 × 2 = 12
```

## リーチ判定

プレイヤーごとに、次にそのプレイヤーが正解した場合にチーム得点が勝利点以上になるかを判定する。

判定式は以下である。

```text
(現在のチーム得点 / 対象プレイヤーの個人得点) × (対象プレイヤーの個人得点 + 1) >= winning-pt
```

リーチ状態のプレイヤーの個人得点セルはオレンジ背景になる。

ただし、封鎖状態の赤背景が優先される。

## 封鎖判定

プレイヤーの誤答数が `✕✕` になった場合、そのプレイヤーは封鎖状態である。

封鎖状態では以下が赤背景になる。

- プレイヤー名セル
- 個人得点セル
- 誤答数セル

封鎖状態のプレイヤーは正解ボタンを押しても得点できない。

## 勝敗判定

再計算時に以下の順番で勝敗を判定する。

### 1. 勝利点到達

チーム得点が `winning-pt` 以上になったチームは勝利する。

- A チームが到達: A 勝利
- B チームが到達: B 勝利

勝利点到達による勝利の場合、勝利チームの得点表示は `winning-pt` に丸めて表示される。

### 2. 全員封鎖

一方のチームの全プレイヤーが `✕✕` になった場合、相手チームが勝利する。

この勝利では、チーム得点は `winning-pt` に丸めない。

### 3. 規定問題数

`infinity` がチェックされている場合、問題数上限による終了判定は行わない。

`infinity` が未チェックの場合、内部問題番号 `secret-counter` が `max-of-questions` を超えたら試合終了判定を行う。

終了時点で、

- A チーム得点 > B チーム得点: A 勝利
- B チーム得点 > A チーム得点: B 勝利
- 同点: 引き分け

### 4. 継続中

上記のいずれにも該当しない場合、試合は継続中である。

## 問題番号の扱い

内部問題番号は `secret-counter` に保持する。

初期値は `1` である。

以下の操作で `+1` される。

- 正解ボタン押下
- 誤答ボタン押下
- `Through` ボタン押下

表示用の `number-of-done` は、カウント表示状態に応じて以下を表示する。

| 状態 | 表示 |
|---|---|
| カウント表示中、規定問題数内 | 現在の内部問題番号 |
| カウント表示中、規定問題数超過 | `END` |
| カウント非表示中 | `?` |
| infinity チェック時 | 内部問題番号を表示し続ける |

## スルー処理

`Through` ボタン押下時の動作は以下である。

1. 試合がすでに終了している場合は何もしない。
2. 問題番号を `+1` する。
3. 履歴表にスルー行を追加する。

スルー行では、各セルの背景色を `#CCCCCC` にする。

現行実装では、スルー時に `CalcAll()` は呼ばれていない。そのため、規定問題数到達による勝敗判定をスルー直後に即時反映したい場合は、差分実装時に要検討である。

## 履歴表

各チームに履歴表がある。

履歴表には以下を表示する。

- 問題番号列 `No.`
- 各プレイヤーの正誤集計 `〇:n ✕:n`
- 問題ごとの正誤履歴

### 履歴追加

正解・誤答・スルーのたびに、両チームの履歴表に 1 行ずつ追加する。

各行の先頭列には問題番号相当の値を表示する。

正解・誤答が発生したチーム側の履歴表だけ、対象プレイヤーの列に以下を表示する。

| 操作 | 表示 |
|---|---|
| 正解 | `〇` |
| 誤答 | `✕` |
| スルー | 各セル背景を灰色にする |

### 履歴集計

履歴表に記録された `〇` と `✕` をプレイヤーごとに集計し、ヘッダーに表示する。

表示形式は以下である。

```text
〇:0 ✕:0
```

### 履歴スクロール

履歴データ部分は縦スクロール可能である。

`syncscroll.js` により、同じ `name` 属性を持つ複数の `.syncscroll` 要素はスクロール位置が同期される。

現行実装では、チーム A / チーム B の履歴エリアに同じスクロールグループ名が設定されており、両チームの履歴スクロールが同期する。

履歴行追加時は、履歴エリアを最下部までスクロールする。

## リセット処理

`reset` ボタン押下時は確認ダイアログを表示する。

確認された場合、以下を行う。

1. 両チーム全プレイヤーの個人得点を `1` に戻す。
2. 両チーム全プレイヤーの誤答数を空欄に戻す。
3. 内部問題番号を `1` に戻す。
4. 表示用問題番号を更新する。
5. 履歴行をすべて削除する。
6. 個人成績集計を初期化する。
7. 勝敗結果表示を空欄に戻す。

## 再計算処理

`refresh` ボタン押下時、または各種操作後に、全体再計算を行う。

再計算では以下を行う。

1. A チームのチーム得点を計算する。
2. B チームのチーム得点を計算する。
3. リーチ判定を行う。
4. 封鎖判定を行う。
5. 勝敗判定を行う。
6. 勝敗表示を更新する。

## 現行実装上の注意点

以下は、既存実装から読み取れる注意点である。差分実装時に修正するか、既存互換として残すかを判断すること。

### 関数名のタイポ

`DisplayMathcResult` は `Match` のスペルミスと思われる。

既存コード互換が不要なら `DisplayMatchResult` に直してよい。

### 戻り値の型が一貫していない

`JudgeMatch()` は、勝敗決定時には配列を返すが、継続中の場合に文字列 `'ongoing'` を返している。

`DisplayMathcResult(result)` は `result[0]` を参照するため、文字列でも偶然動くが、明確な設計としては戻り値を統一した方がよい。

推奨例：

```js
{ status: 'ongoing', reachedWinningPoint: false }
```

### グローバル変数化している箇所

`ConvertNumberDoubleToSingleByte` や `NormalizeAndValidateNumberString` 内の `ret` が `let` / `const` なしで代入されている。

新規実装では `let` または `const` を使うこと。

### スルー時の勝敗判定

`Through` ボタンでは問題数を進め、履歴を追加するが、`CalcAll()` は呼ばれていない。

規定問題数超過による終了判定をスルー直後に反映したい場合は、`CalcAll()` を呼ぶ必要がある。

### リセット時の再計算回数

`ResetPlayerPt()` 内でプレイヤーごとに `CalcAll()` を呼んでいる。

新規実装では、全プレイヤーを初期化したあとに 1 回だけ再計算すればよい。

### HTML に状態を持ちすぎている

現行実装は、得点・誤答・履歴・問題番号などの状態を DOM の `innerText` に直接持たせている。

新規実装では、可能であれば JavaScript オブジェクトとして状態を管理し、DOM は表示結果として描画する方が望ましい。

ただし、画面仕様を再現する段階では DOM ベースの実装でもよい。

## 新しい Web ページを作るときの推奨設計

差分指示で大きな変更が入る可能性があるため、新規実装では以下の分離を推奨する。

### 状態

最低限、以下の状態を持つ。

```js
const state = {
  winningPoint: 200,
  maxQuestions: 40,
  infinity: false,
  currentQuestion: 1,
  showCount: true,
  showHistory: true,
  teams: [
    {
      id: 'a',
      name: 'A',
      players: [
        { id: 'a1', name: 'A1', point: 1, incorrect: 0 },
        { id: 'a2', name: 'A2', point: 1, incorrect: 0 },
        { id: 'a3', name: 'A3', point: 1, incorrect: 0 },
        { id: 'a4', name: 'A4', point: 1, incorrect: 0 },
        { id: 'a5', name: 'A5', point: 1, incorrect: 0 }
      ]
    },
    {
      id: 'b',
      name: 'B',
      players: [
        { id: 'b1', name: 'B1', point: 1, incorrect: 0 },
        { id: 'b2', name: 'B2', point: 1, incorrect: 0 },
        { id: 'b3', name: 'B3', point: 1, incorrect: 0 },
        { id: 'b4', name: 'B4', point: 1, incorrect: 0 },
        { id: 'b5', name: 'B5', point: 1, incorrect: 0 }
      ]
    }
  ],
  history: []
};
```

### 履歴

履歴は以下のようなデータとして持つと扱いやすい。

```js
{
  questionNo: 1,
  type: 'correct', // 'correct' | 'incorrect' | 'through'
  teamId: 'a',
  playerId: 'a3'
}
```

スルーの場合は `teamId` と `playerId` を `null` にする。

### 描画

状態を更新したら、以下を再描画する。

1. チーム得点
2. 個人得点
3. 誤答表示
4. リーチ表示
5. 封鎖表示
6. 勝敗表示
7. 問題番号表示
8. 履歴表
9. 個人別正誤集計

## 実装時の確認観点

Codex は変更後、少なくとも以下を確認すること。

- 初期表示で両チーム得点が `1` になること。
- 正解ボタンで個人得点が `+1` されること。
- 誤答ボタンで誤答数が進み、個人得点が `1` に戻ること。
- `✕✕` のプレイヤーが正解できないこと。
- 誤答時、相手チームの `✕✕` が `✕` に戻ること。
- チーム得点が個人得点の積になっていること。
- 勝利点到達で `WIN` / `LOSE` が表示されること。
- 全員 `✕✕` で相手チームが勝利すること。
- 規定問題数超過時、得点の高いチームが勝利すること。
- 同点終了時、両チーム `DRAW` になること。
- `infinity` チェック時、問題数上限で終了しないこと。
- `Hide Count` / `Show Count` で問題番号と履歴番号の表示が切り替わること。
- `Hide History` / `Show History` で履歴エリアが切り替わること。
- `reset` で状態と履歴が初期化されること。

## ユーザーに確認した方がよい差分ポイント

改修の際、以下はユーザーの意図により変わる可能性が高い。

- 2 チーム固定か、チーム数を増減可能にするか。
- 1 チーム 5 人固定か、人数を変更可能にするか。
- AQL ルールを維持するか、別ルールにするか。
- チーム得点を「個人得点の積」のままにするか。
- 誤答 `✕✕` の封鎖ルールを維持するか。
- 相手チーム誤答時に封鎖が 1 段階戻るルールを維持するか。
- 履歴表を表示するか。
- 操作用画面と観客表示用画面を分けるか。
- スマホ・タブレット対応を重視するか。
- Google Analytics を残すか削除するか。
